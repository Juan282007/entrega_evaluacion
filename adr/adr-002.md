# ADR-002: Manejo de roles con permisos diferenciados para seguridad y trazabilidad

## Fecha
2026-04-14

## Estado
Aceptado

## Contexto
El modelo ya tiene tablas de seguridad como `security_role`, `security_permission`, 
`user_account`, `user_role` y `role_permission`. Sin embargo, no existe una estrategia 
definida de qué roles existen, qué pueden hacer y cómo se controla quién asigna permisos.

## Problema
Sin roles y permisos definidos, cualquier usuario del sistema podría acceder a 
operaciones críticas como procesar pagos, modificar tarifas o validar abordajes. 
Esto representa un riesgo de seguridad y dificulta saber quién hizo qué dentro del sistema.

## Decisión
Se usarán las entidades existentes del modelo para definir roles concretos con 
permisos específicos por operación:

- `ROLE_ADMIN` — acceso total
- `ROLE_AGENT` — reservas y check-in
- `ROLE_FINANCE` — pagos y facturación
- `ROLE_OPS` — operaciones de vuelo
- `ROLE_READONLY` — solo consulta

## Justificación técnica
- No se necesitan nuevas tablas, el modelo ya tiene todo lo necesario.
- El campo `assigned_by_user_id` en `user_role` permite saber quién asignó cada rol.
- Los roles y permisos se cargarán mediante scripts DML versionados en Liquibase.
- Es una práctica estándar en sistemas con múltiples tipos de usuarios.

## Consecuencias
- Se poblarán las tablas `security_role` y `security_permission` con datos reales.
- El equipo documentará la matriz de roles y permisos.
- Las operaciones críticas quedarán protegidas por perfil de usuario.
- Mejora la trazabilidad y el control de acceso en todo el sistema.
