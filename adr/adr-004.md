# ADR 004: Definir estrategia de versionamiento del repositorio con ramas develop, qa y main

## Fecha

2026-04-14

## Estado

Aceptado

## Contexto

El proyecto requiere una estrategia de branching que permita desarrollo colaborativo, pruebas en un entorno intermedio y despliegue controlado a producción.

## Decisión

Adoptar un flujo de Git con las siguientes ramas principales:

- `develop`: integración continua de características en curso y base para pruebas internas.
- `qa`: rama destinada a validación de calidad y pruebas antes de liberar a producción.
- `main`: rama de producción, solo con versiones estables y aprobadas.

### Reglas

- Las nuevas funcionalidades se desarrollan en ramas de características (`feature/...`) a partir de `develop`.
- Las correcciones de bugs se integran en `develop` y, una vez validadas, se promueven a `qa`.
- `qa` se usa para pruebas de integración y verificación de release candidates.
- Solo se permite merge a `main` desde `qa` tras aprobación.

## Razones

- Separa claramente el desarrollo activo del código preparado para pruebas y del código de producción.
- Facilita la colaboración entre desarrolladores y equipos de QA.
- Reduce el riesgo de desplegar cambios no validados.
- Permite establecer políticas de revisión y aprobaciones en cada etapa.

## Consecuencias

- Se requiere disciplina para mantener `develop` actualizada y `qa` alineada con los criterios de prueba.
- Los merges a `main` serán eventos deliberados y controlados.
- El equipo debe documentar el flujo y usar pull requests para cada cambio.
- Se mejora la estabilidad del repositorio y la trazabilidad de releases.
