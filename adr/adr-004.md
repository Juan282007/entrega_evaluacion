# ADR-004: Estrategia de versionamiento del repositorio con ramas develop, qa y main

## Fecha
2026-04-14

## Estado
Aceptado

## Contexto
El proyecto necesita una forma ordenada de trabajar en equipo, probar cambios 
antes de publicarlos y mantener una versión estable del código en todo momento.

## Problema
Sin una estrategia de ramas definida, los cambios pueden mezclarse sin control, 
es difícil saber qué está listo para producción y cualquier error puede afectar 
directamente el entorno estable del proyecto.

## Decisión
Se adoptará un flujo de trabajo con tres ramas principales:

- `develop` — rama de trabajo diario donde se integran los cambios en curso
- `qa` — rama para validar cambios antes de publicarlos
- `main` — rama estable, solo recibe cambios aprobados desde qa

El flujo de trabajo es:
1. Cada historia de usuario se desarrolla en una rama propia (`HU-XX-dev`)
2. Se hace Pull Request de `HU-XX-dev` hacia `develop`
3. Cuando hay un conjunto estable, se hace PR de `develop` hacia `qa`
4. Solo después de validar en `qa` se hace PR hacia `main`

## Justificación técnica
- Separa claramente el trabajo en curso del código validado y del código estable.
- Los Pull Requests permiten revisar cambios antes de integrarlos.
- Reduce el riesgo de publicar cambios con errores.
- Es un flujo estándar usado en proyectos profesionales de desarrollo.

## Consecuencias
- Todo el trabajo se hace desde `develop`, nunca directamente en `main` o `qa`.
- Los merges a `main` son eventos controlados y deliberados.
- El equipo debe mantener `develop` actualizada antes de crear nuevas ramas.
- Se mejora la estabilidad del repositorio y la trazabilidad de cambios.
